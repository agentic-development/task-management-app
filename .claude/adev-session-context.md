## Identity

The Agentic Development Platform is a spec-driven agent orchestration, context engineering, and governance platform. It integrates Feature Charters, Live Specs, Context Index, Agent Task Routing, AgentOps Dashboard, and Eval Harness into a single operational surface for teams adopting AI-assisted software delivery.

## Non-Negotiable Principles

1. **Event-driven boundaries** — Bounded contexts communicate via domain events only. Direct cross-context API calls are limited to read-only queries. No context may import internal modules from another context.
2. **Model-agnostic orchestration** — No AI provider-specific abstractions in the domain layer. Model details (API keys, prompt formats, token counting) stay at the integration boundary.
3. **Spec-first development** — The product charter and approved specs under `.context-index/specs/` are the source of truth. Code implements what specs define; unspecified behavior requires a spec update before implementation.
4. **Dual-layer storage** — Context artifacts (specs, charters, ADRs, governance policies) are stored as files in Git for versioning, diffing, and auditability. Runtime state (sessions, projects, users, config) is persisted to SQLite for crash recovery and indexed queries. See ADR-0005.
5. **Observable by default** — All domain operations emit OpenTelemetry traces via the InstrumentedEventBus. No silent side effects; every meaningful state change is traceable.

## Coding Standards

### Language and Runtime

- **Primary language:** TypeScript (strict mode)
- **Runtime:** Node.js (ES2022 target, ESM modules)
- **Package manager:** npm

### Conventions

- **Naming:** camelCase for variables/functions, PascalCase for types/interfaces/classes, UPPER_SNAKE_CASE for constants
- **File structure:** One primary export per file, kebab-case filenames
- **Imports:** Use path aliases (`@spec-management/*`, `@context-engine/*`, `@orchestration/*`, `@eval-governance/*`, `@observability/*`, `@integration/*`, `@web/*`, `@shared/*`) — never relative cross-context imports
- **Error handling:** Domain errors are typed; throw specific error classes, not generic `Error`
- **Logging:** Use the shared Logger from `@shared/logger` which wraps OpenTelemetry

### Patterns to Follow

- Event envelope pattern for all domain events (see `src/shared/events.ts`)
- State machines for lifecycle transitions (spec states, session states)
- Interface-first design: define the contract in `@shared/`, implement in the bounded context

### Anti-Patterns to Avoid

- Importing internal modules across bounded context boundaries
- Provider-specific types in domain interfaces (e.g., `OpenAIResponse` in orchestration)
- Mutable shared state between contexts (use events instead)
- Synchronous cross-context calls for write operations

### Golden Sample: API Route Handler Pattern
# Golden Sample: API Route Handler Pattern

This sample demonstrates the canonical API route handler pattern for the Next.js App Router layer (`app/api/`). Follow this structure when adding new API endpoints.

## Route Handler

```typescript
// app/api/<resource>/route.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";
import { getApplication } from "../../lib/app-singleton";

// 1. Define response type interface (co-located, not in a shared types file)
interface ResourceSummary {
  id: string;
  title: string;
  status: string;
  createdAt: string;
}

// 2. GET handler — list resources
export async function GET(): Promise<NextResponse<ResourceSummary[]>> {
  const app = await getApplication();
  const items = await app.someContext.listResources();

  const summaries: ResourceSummary[] = items.map((item) => ({
    id: item.id,
    title: item.title,
    status: item.status,
    createdAt: item.createdAt,
  }));

  return NextResponse.json(summaries);
}

// 3. POST handler — create resource (with request body validation)
export async function POST(request: NextRequest): Promise<NextResponse> {
  const body = await request.json();

  // Minimal validation at the API boundary
  if (!body.title || typeof body.title !== "string") {
    return NextResponse.json(
      { error: "title is required" },
      { status: 400 },
    );
  }

  const app = await getApplication();
  const created = await app.someContext.createResource({
    title: body.title,
  });

  return NextResponse.json(created, { status: 201 });
}
```

## Dynamic Route Handler

```typescript
// app/api/<resource>/[id]/route.ts
import { NextResponse } from "next/server";
import { getApplication } from "../../../lib/app-singleton";

// Next.js 16: params is async
export async function GET(
  _request: Request,
  { params }: { params: Promise<{ id: string }> },
): Promise<NextResponse> {
  const { id } = await params;
  const app = await getApplication();

  try {
    const item = await app.someContext.getResource(id);
    return NextResponse.json(item);
  } catch (error) {
    // Domain errors map to HTTP status codes
    if (error instanceof Error && error.message.includes("not found")) {
      return NextResponse.json(
        { error: `Resource "${id}" not found` },
        { status: 404 },
      );
    }
    return NextResponse.json(
      { error: "Internal server error" },
      { status: 500 },
    );
  }
}
```

## Key Rules

1. **Named HTTP method exports** — `GET`, `POST`, `PUT`, `PATCH`, `DELETE` as named exports
2. **Return type annotations** — always annotate with `Promise<NextResponse<T>>` for type safety
3. **getApplication()** — access domain services through the singleton, never import context internals directly
4. **Response interfaces** — define co-located with the route, not in a shared types file (projections are route-specific)
5. **Validate at boundary** — check required fields in the route handler; domain services handle business rules
6. **Error mapping** — catch domain errors and map to appropriate HTTP status codes (404, 400, 500)
7. **Async params** — in Next.js 16, dynamic route `params` are a `Promise` and must be awaited
8. **No raw data exposure** — map internal entities to response projections (strip internal fields)
9. **JSON responses** — use `NextResponse.json()` for all responses, including errors


### Golden Sample: Event Subscriber Pattern
# Golden Sample: Event Subscriber Pattern

This sample demonstrates the canonical event subscription pattern used by bounded contexts to react to domain events from other contexts. Follow this structure when adding new cross-context event handlers.

## Event Handler Registration

```typescript
// src/<context>/event-handlers.ts
import type { EventBus } from "@shared/event-bus.js";
import type { DomainEvent } from "@shared/events.js";

// 1. Declare event types this handler cares about as a const array
const SPEC_EVENT_TYPES = [
  "spec.created",
  "spec.updated",
  "spec.status-changed",
  "charter.created",
  "charter.updated",
] as const;

// 2. Export a single registration function — called from app bootstrap wiring
export function registerEventHandlers(
  eventBus: EventBus,
  state: ContextState,
): void {
  // 3. Subscribe to each event type in a loop
  for (const eventType of SPEC_EVENT_TYPES) {
    eventBus.subscribe(eventType, (event) => {
      // 4. Handle the event — update local state, trigger side effects
      state.timestamps["live-specs"] = new Date(event.timestamp);
    });
  }
}
```

## Event Handler with Side Effects (Observability Pattern)

```typescript
// src/<context>/event-handlers.ts
import type { EventBus } from "@shared/event-bus.js";
import type { DomainEvent } from "@shared/events.js";
import { createLogger } from "@shared/logger.js";

const logger = createLogger("my-context-events");

// For contexts that need ALL events (like Observability), subscribe broadly
export function registerAllEventHandler(
  eventBus: EventBus,
  store: AggregateStore,
): void {
  // Subscribe to specific event categories
  const sessionEvents = [
    "session.started",
    "session.completed",
    "session.paused",
    "session.resumed",
  ] as const;

  for (const eventType of sessionEvents) {
    eventBus.subscribe(eventType, (event: DomainEvent) => {
      try {
        store.processEvent(event);
      } catch (error) {
        // 5. Log but don't rethrow — subscriber errors must not break the publisher
        logger.warn("Failed to process event", {
          eventType: event.type,
          eventId: event.id,
          error: error instanceof Error ? error.message : String(error),
        });
      }
    });
  }
}
```

## Wiring in App Bootstrap

```typescript
// src/app/wiring.ts (excerpt)
import { registerEventHandlers } from "@context-engine/event-handlers.js";

export function wireEventSubscriptions(
  eventBus: EventBus,
  services: Services,
): void {
  // 6. Register all event handlers during application bootstrap
  registerEventHandlers(eventBus, services.contextEngine.getIndexState());

  // Each context registers its own handlers
  registerObservabilityHandlers(eventBus, services.observability);
  registerSSEHandlers(eventBus, services.web.getBroadcaster());
}
```

## Key Rules

1. **Const event type arrays** — declare subscribed events as `as const` arrays for type safety and discoverability
2. **Single registration function** — one `registerEventHandlers()` export per context, called from `src/app/wiring.ts`
3. **No cross-context imports** — import only from `@shared/*`, never from other bounded contexts
4. **Subscriber error isolation** — catch errors in handlers, log them, never rethrow. Subscriber failures must not break the publishing context's write path
5. **Minimal handler logic** — handlers should update local state or trigger local operations. Complex workflows go in service methods, not event handlers
6. **Event envelope access** — all events conform to `EventEnvelope<T, P>` with `id`, `type`, `timestamp`, `source`, `payload`
7. **Wiring is centralized** — `src/app/wiring.ts` is the single place where all event subscriptions are registered
8. **No event persistence** — events are in-memory only (ADR-0002). If you need history, maintain pre-computed aggregate stores like Observability does

// 1. Export props interface separately for reuse
export interface KpiCardProps {
  label: string;
  value: string;

const trendIndicator: Record<string, { symbol: string; color: string }> = {

## Key Rules

1. **Server Components by default** — no `'use client'` unless browser APIs or interactivity is needed
2. **Metadata export** — every page exports `metadata` for SEO/title
3. **Partial failure resilience** — declare defaults, wrap data fetch in try/catch, render what you have
4. **getApplication()** — always `await` the singleton; it initializes lazily on first call
5. **Dark mode tokens** — use `zinc-800/900` borders and backgrounds, `zinc-400/500` text
6. **shadcn/ui primitives** — compose from Card, Table, Badge, etc.; don't build raw HTML
7. **EmptyState** — always handle the zero-items case with an icon and message
8. **Named exports** — use `export function` not `export default function` for non-page components
9. **Lookup tables** — prefer `Record<string, Style>` over chains of ternaries for variant styling
10. **Font mono** — use `font-mono` for IDs, timestamps, tokens, and any machine-readable values


### Golden Sample: Bounded Context Service Pattern
# Golden Sample: Bounded Context Service Pattern

This sample demonstrates the canonical service pattern used across all bounded contexts. Follow this structure when adding new services or methods.

## Service Class

```typescript
// src/<context>/index.ts
import { randomUUID } from "node:crypto";
import type { EventBus } from "@shared/event-bus.js";
import type { DomainEvent } from "@shared/events.js";

export class ExampleService {
  constructor(
    private readonly repo: ExampleRepository,
    private readonly eventBus: EventBus,
  ) {}

  async createEntity(input: CreateInput): Promise<SharedEntity> {
    // 1. Validate inputs (throw typed domain errors)
    if (!input.name) {
      throw new ValidationError(["name is required"]);
    }

    // 2. Build internal entity
    const entity: InternalEntity = {
      id: randomUUID(),
      name: input.name,
      createdAt: new Date().toISOString(),
    };

    // 3. Persist (if this throws, no event is emitted)
    await this.repo.save(entity);

    // 4. Emit event (catch subscriber errors — don't break the write path)
    this.safePublish({
      id: randomUUID(),
      type: "entity.created",
      timestamp: entity.createdAt,
      source: "example-context",
      payload: { entityId: entity.id, name: entity.name },
    });

    // 5. Return shared projection (strip internal fields)
    return toSharedEntity(entity);
  }

  private safePublish(event: DomainEvent): void {
    try {
      this.eventBus.publish(event);
    } catch {
      // Subscriber errors must not break the write flow
    }
  }
}
```

## Key Rules

1. **Write flow**: validate → persist → emit (never emit before persist succeeds)
2. **Typed errors**: always throw specific `DomainError` subclasses with codes
3. **Safe publish**: catch subscriber errors in `safePublish()` wrapper
4. **Interface-first**: repositories are interfaces, Git-backed implementations
5. **Shared projection**: internal entities project to minimal shared types for cross-context consumption
6. **TDD**: write failing test first, implement to pass, commit
